# Foundry CLI UX Guidelines

This document defines the baseline UX rules for Foundry CLI commands.

If you are implementing a new CLI command, you should follow this document
and treat it as a contract.

It is intentionally small. The goal is consistency, not bureaucracy.

Remember the core tenets of Foundry: **simplicity and determinism above all.**

## Command shape

Foundry commands should follow this structure:

```bash
foundry <command> <subcommand> [options]
```

Examples:

```bash
foundry version
foundry plugin list --enabled
foundry theme scaffold editorial
foundry routes check
```

## Output modes

Commands should generally fall into one of these output styles:

### Tabular Output

These should prefer aligned tabular output when showing multiple rows.

Existing commands that follow this practice:

```bash
foundry plugin list --enabled 
foundry plugin list --installed
foundry theme list
foundry routes list
foundry content list
```

### Marker Output

These should use explicit status markers where appropriate.

Preferred markers include `[OK]` and `[FAIL]`.

Existing commands that follow this practice:

```bash
foundry plugin validate
foundry doctor
foundry validate
foundry routes check
```

### Mutators

These should clearly state what changed, and print `Next steps:` when
user action is still required.

Existing commands that follow this practice:

```bash
foundry plugin install
foundry plugin enable
foundry plugin disable
foundry theme switch
```

## Preferred wording

### Use `foundry ...` not `make ...`

User facing command suggestions should prefer the public CLI tool over the Makefile, with wording like:

> Run `foundry plugin sync`

instead of:

> Run `make plugins-sync`

`make` remains a developer convenience, **_not the primary user interface_**.

### Keep success lines direct

    Enabled plugin: toc
    Switched theme to "default"
    plugin imports synced
    build complete

### Keep failure messages direct

    plugin not found: seo
    route collision: "/about/" is generated by both ...
    theme "editorial" is missing theme.yaml

## Next steps format

When a command mutates project state and the user must do something else
afterward, print an action list:

    Next steps:
    1. ...
    2. ...

**Do not** print `Next steps:` for purely read-only commands.

## Config requirements

Commands should not require project config unless they actually need it.

An example of config-free commands is `foundry version`.

An example of config-bound commands is `foundry build`.

## Validation scope

Validation style commands should be clear about scope:

1. config check validates config only
2. plugin validate validates plugins
3. routes check validates routes
4. validate validates the overall project
5. doctor checks project + environment health

Commands should not silently expand scope beyond what their name
implies. KEEP IT SIMPLE.

## Human-first output

Default CLI output should be optimized for humans.

Machine-readable output such as `--json` may be added, but
human-readable output remains the default. Foundry should stay accessible
to both developers and non-developers.

## Incremental adoption

This contract applies to all new commands and to existing commands whenever they are modified.
